# QuadShot — FPV Flight Sandbox

3D FPV drone flight sandbox in Godot, growing into an action flight game. (Game title: **QuadShot** — one o; the repo folder keeps its historical `QuadShoot` spelling.) **The flight model is the product** — everything serves the human-in-the-loop tuning cycle. Milestone 0 spec: [FPV-SANDBOX-HANDOFF.md](FPV-SANDBOX-HANDOFF.md); everything after: [ROADMAP.md](ROADMAP.md); the M6 gameplay model (living-theater campaign) is designed in [GAMEPLAY-DESIGN.md](GAMEPLAY-DESIGN.md) — a living doc + decision log, **append-only** (history is never pruned). Read the relevant one before big changes.

## Running & verifying

- Godot 4.7 stable (standard/GDScript build, portable): `C:\Tools\Godot\Godot_v4.7-stable_win64_console.exe` (use the `_console` exe from CLI so output is visible).
- Run the game: `<exe> --path .`
- Run the **dev room** testbed: `<exe> --path . scenes/dev_map.tscn` — a big sandbox map that mirrors main's wiring and gets a specimen of every game element as they're added (shooting range, city block, slalom, tunnel, platforms, crash wall).
- **Look pass** (post/atmosphere + emissive), applied game-wide (both greybox and dev_room): a `LookController` (`scripts/environment/look_controller.gd`) applies `LookConfig` (`resources/look_config.gd`, AgX tonemap + glow/SSAO/fog/color-grade/sun) onto the scene's Environment/Sun every frame, live-tuned via the overlay's **LOOK** section. Emissive palette by role (bloom threshold is 1.0, so accents use energy >1): cyan/blue = navigation (gates), red = threat (enemies/turrets), orange = score (targets), amber = course pylons, green = pads, yellow = your fire. Ground is `checker_ground.gdshader` (neon grid); structures use `neon_structure.gdshader` (panel seams). The overlay leaves `look_config` null → LOOK section skipped only where a scene has no LookController.
- Headless verify after edits (treat warnings as errors):
  - Re-import: `<exe> --headless --import --path .`
  - Boot check: `<exe> --headless --quit-after 10 --path .`
  - Script parse check: `<exe> --headless --check-only -s <script.gd> --path .`

## Architecture

- `scenes/` — `main.tscn` (entry: environment + drone + combat + UI), `dev_map.tscn` (dev-room testbed, same wiring), `environment/` (greybox, dev_room), `drone/drone.tscn`, `combat/` (projectile, explosion, target, turret), `ui/` (debug_overlay, hud, draft_screen)
- `scripts/` — `drone/` (flight_controller, motor_model, input_handler, rate_controller), `combat/` (weapon, projectile+pool, health, turret, target, enemy_drone, wave_director, effects), `war/` (M6 war-sim: theater_generator + war_sim — pure static-func modules over a serializable state Dictionary; deterministic, seed-driven; state round-trips var_to_str bit-exactly = the portable save; benches: `tests/war_soak.gd` 200-theater invariant soak, `tests/war_trace.gd` single-war trace), `audio/` (sound_bank — synthesized, no external assets; motor/wind emitters), `ui/`, `tests/` (headless checks: hover, combat, wave)
- `resources/` — `tunable_config.gd` base + `flight_config.gd`/`combat_config.gd`/`damage_config.gd`/`enemy_config.gd` and their `default_*.tres` (shared instances: every exporter of the same `.tres` sees live edits), shared shaders/materials. **`EnemyConfig` is one class with many instances** (`default_enemy_<type>.tres`, one per bestiary type per GAMEPLAY-DESIGN P4.8), so its save/defaults paths derive from `type_id` instead of being class constants; `CombatConfig` is player-side only
- Combat wiring: projectiles call `take_hit(damage)` on whatever they hit unless it shares the shooter's `team`; entities award score via a `destroyed(points)` signal that `main.gd` tallies; `SoundBank` is a scene node with a null-safe **static** API (not an autoload — autoloads don't exist under `--script` test runs).
- Physics tick is **240 Hz** (rate-controller stability); all flight code runs on the fixed tick.
- **Pilot axis convention** for all Vector3 rate/gain fields: x = roll (+right), y = pitch (+nose up), z = yaw (+right). Drone front is body **-Z**. Conversion from Godot space happens only in `FlightController._measured_rates()`.

## Conventions

- **No magic numbers in physics/input code** — every tunable is an `@export` in `FlightConfig`; scripts read it, the debug overlay writes it live.
- Static typing everywhere; `snake_case`; short focused functions; comments explain *why*, not *what*.
- Composition over inheritance; scenes minimal; `.tscn`/`.tres`/`project.godot` are hand-edited text — keep them small and diff-readable.
- Hand-written `.tscn` gotcha: exported Node properties need `node_paths=PackedStringArray("prop")` on the `[node]` line **and** `prop = NodePath("...")` in its body, or the reference silently loads as Nil.
- Never touch `.godot/`. No third-party addons or external assets without asking — greybox primitives and flat/procedural materials only.
- Explain non-obvious reasoning briefly; flag uncertainty instead of guessing silently.

## Controls & tuning

- Gamepad (Mode 2): left stick throttle/yaw, right stick pitch/roll. **A** arm (throttle at zero-thrust position), **B** reset, **Y** acro/angle mode, **X** FPV/chase camera, **RT** fire blaster, **LT** fire missile (requires lock: hold an enemy near the reticle until the diamond turns red), **Start** debug overlay. Keyboard fallbacks ship by default (Enter arm, R reset, M mode, C camera, Space fire, F missile, Tab overlay).
- **Input system**: `input_profile` dropdown (top of overlay FLIGHT) selects gamepad vs USB radio (EdgeTX joystick mode, e.g. RadioMaster TX16S; AETR/TAER channel orders on axes 0–3; auto-detected by joypad name, gamepad fallback; wrong-direction axes get flipped on the radio's Outputs page). Buttons/switches bind via the overlay's **BINDINGS** section: `InputBindings` (`resources/input_bindings.gd`) rewrites the InputMap at runtime — press-to-bind captures keys, joypad buttons, **and axes as switches** (how radio switches arrive). `arm_switch` is stateful FPV arming (switch position = armed), unbound by default; `arm_toggle` stays momentary; `missile_auto_switch` (also stateful, unbound) is the missile director — with it on, a stable full lock auto-launches after `missile_auto_hold_s`. The blaster's gun director is `fire_assist_miss_m` (CombatConfig; 0 = off). Persists to `user://input_bindings.tres` via the normal Save/Load/Defaults.
- **Pause mode** (BeamNG-style): `pause_toggle` (default **P**) / `pause_switch` slow time to `pause_time_scale` (FlightConfig) instead of stopping; the drone's autopilot holds position (level-and-brake + hover collective, `autopilot_*` tunables) and bindings swap to the **paused context** (second mapping set, gameplay actions unbound by default — safe typing in overlay fields; edit via the BINDINGS context dropdown). Pause/overlay actions always stay bound from the flight set.
- **WEATHER (todo)** overlay section: `WeatherConfig` stub (wind/precipitation/fog/heat fields, persisted but not yet simulated) — placeholder for GAMEPLAY-DESIGN P1.6.
- Headless test suite under `scripts/tests/`: `hover_check.gd`, `combat_check.gd`, `wave_check.gd`, `missile_check.gd`, `run_check.gd` — run all after flight/combat changes.
- **Balance instrument** (GAMEPLAY-DESIGN v1.23/v1.24) — read [BALANCE.md](BALANCE.md) before touching it or acting on its output. Three layers, one command (`tools/balance_report`): **Layer 1 lethality** (`scripts/balance/lethality.gd`, config arithmetic; verified by `tests/lethality_check.gd` planting shots into a real `Health`), **Layer 2 delivery** (`tests/delivery_bench.gd` — aim vs a static target, evasion vs a frozen perfect shooter; writes `balance/delivery_factors.json`), and **validation** (`tests/matchup_harness.gd`, which prints the mini-web as paper → predicted → validated). `ReferencePilot.PILOT_VERSION` pins the measuring brain — bump it on ANY pilot behavior change, then re-measure deliberately; factors from another pilot version blank the predicted column instead of mixing rulers. A predicted-vs-validated gap is the instrument's *output* (it names an un-modeled factor), not a number to tune away.
- Rate-loop wobble is fought with the FlightConfig **Filtering** group (gyro LPF, D-term LPF, RC smoothing — Betaflight-style), never by changing the physics.
- Rate feel is a `rate_preset` dropdown at the top of the overlay's FLIGHT section (`FlightPresets`, `scripts/drone/flight_presets.gd`): Default/Cinematic/Freestyle/Race, each an atomic snapshot of rate_p/i/d/ff + angular_damping. Hand-tuning any of those five fields falls through to "Custom" — detected live by comparing the config against every preset, not tracked as separate state. Bench-tune new presets with `scripts/tests/rate_tune_sweep.gd` and `step_response.gd` before adding them.
- Run flow (M4): arming starts a run of sorties (`sortie_waves` waves each); clearing a sortie opens the exit gate, and flying it triggers a paused upgrade draft applied to the run-scoped `RunMods` static layer (never to the configs). Death ends the run, records `user://profile.json`, shows the summary; the next arm starts fresh (`RunMods.reset()`). Score is run-scoped with a combo multiplier.
- Overlay: mouse tunes every `FlightConfig` field live while the gamepad flies. Sections (FLIGHT/COMBAT/BESTIARY/DAMAGE/AUDIO/BINDINGS/LOOK — one BESTIARY block per enemy type) are collapsible (FLIGHT starts open); each has a **preset bar** saving/loading/deleting that group's values as named `.tres` under `user://presets/<group>/`. Save/Load persists to `user://flight_config.tres` (auto-loaded on startup); Defaults re-reads `default_flight_config.tres`. New defaults get baked into that `.tres` only when the human says the feel is right (handoff §14).

## Checkpoint protocol

Work is phased (handoff §11). **At each phase checkpoint: stop, summarize what changed and why, and wait for the human to run/fly it.** Flight feel cannot be evaluated by the agent — the human's hands are the test suite. Map feel feedback to config changes per handoff §14.
